# TypeInference / ASTInserter 実装ガイド

## このガイドの目的

`TypeInference.fs` と `ASTInserter.fs` は、いったん空の型だけを残した状態になっています。
ここから再実装するときは、以前の大きな実装をまとめて戻すのではなく、責務を分離しながら小さく積み上げるほうが安全です。

このガイドでは、次の 3 点を明確にします。

1. 何から順番に作るべきか
2. 各段階でどこまで実装すればよいか
3. どの段階で何を確認すれば壊れにくいか

## まず最初に考えること

型推論を実装する前に、`Parser` と `FlatAST` が何を保証しているかを固定してください。
型推論は AST 構造への依存が強いため、ここが曖昧なままだと後から必ず崩れます。

特に先に整理したいのは次の点です。

- `ASTNode.Type` にどのような種類があるか
- `ASTNode.Data` の各要素が何を意味するか
- ノード参照と即値がどのように区別されているか
- `-1` や `-2` のような特別な参照値があるか
- `type` 系ノードがどういう階層構造を取るか
- `let`、`func`、`func_impl`、`operand_func` が型情報をどこに持つか

この整理が終わるまでは、複雑な推論ロジックを入れないほうがよいです。

## 推奨する全体順序

実装は次の順番で進めるのがおすすめです。

1. AST 契約を確認し、最低限の仕様メモを作る
2. `ASTInserter` を先に再実装する
3. `TypeInference` の内部型表現を作る
4. AST を読むだけの宣言収集パスを作る
5. 式単位の型推論を小さく追加する
6. 推論結果の AST 書き戻しを実装する
7. 重複排除や最適化は最後に入れる

重要なのは、AST の書き換えより先に「正しく読む」能力を作ることです。
読む処理と書く処理を同時に進めると、どちらの不具合なのか切り分けにくくなります。

## フェーズ 1: AST 契約の明文化

### 目標

型推論器が前提にする AST 仕様を固定することです。

### やること

- `Parser.fs` と `FlatAST.fs` を読み、型に関係するノード種別を列挙する
- 各ノードについて `Data` の並び順を表にする
- 子ノード参照を持つノードと、即値しか持たないノードを分ける
- 特殊値の意味を明記する

### 最低限整理したいノード

- `type`
- `type_s`
- `type_func`
- `type_g`
- `type_abs`
- `type_bw_mut`
- `type_bw_&`
- `type_bw_&mut`
- `let`
- `func`
- `func_impl`
- `frame`
- `field`
- `ident`
- `call_func`
- `operand_func`
- `if`
- `operator_*`

### 仕様メモの例

少なくとも次のようなメモがあると実装しやすくなります。

```text
func = [isPub, name, returnTypeRef, parameters, body]
let  = [isMut, isRepo, name, typeRef, content]
```

この情報がないまま推論を書き始めると、後から parser 側の仕様変更に追従しにくくなります。

## フェーズ 2: ASTInserter を先に再実装する

### 先に作る理由

`TypeInference` が正しく推論できても、AST への挿入と参照更新が壊れていたら結果は使えません。
一方で `ASTInserter` は責務が比較的狭く、単体で検証しやすいです。
そのため、まずここを安定させると後の切り分けがかなり楽になります。

### ASTInserter の責務

`ASTInserter` の責務はできるだけ限定してください。

持たせる責務:

- 指定位置へのノード挿入
- 複数ノードの一括挿入
- 既存ノードの置換
- 挿入に伴う参照番号の更新
- `FlatAST` への反映

持たせない責務:

- 型推論そのもの
- 型ノードの意味解析
- どのノードに何型を挿入すべきかの判断
- 重複型ノードの統合戦略

### 推奨 public API

```fsharp
type ASTInserter(fast: FlatAST) =
    member insert: int -> ASTNode -> unit
    member insertMany: int -> ASTNode list -> unit
    member replace: int -> ASTNode -> unit
    member commit: unit -> unit
    member getAST: unit -> ASTNode array
```

### 実装方針

- 内部に mutable な作業用 `ASTNode array` を持つ
- `insertMany` を本体にして `insert` は薄いラッパーにする
- `replace` は配列長を変えずにノードだけ差し替える
- `commit` でのみ `FlatAST` に反映する
- 参照更新は private helper 1 箇所に閉じ込める

### 難所

もっとも壊れやすいのは「途中にノードを差し込んだとき、既存の参照番号がすべて後ろにずれる」点です。
このロジックが複数箇所に散ると、ほぼ確実に整合性が崩れます。
そのため、参照シフトは必ず単一関数に集約してください。

### この段階で書くべきテスト

- 先頭への 1 ノード挿入
- 中間への複数ノード挿入
- 末尾への挿入
- `ref:` を含む既存データのシフト
- `replace` が他ノード参照を壊さないこと
- `commit` 後に `FlatAST` の再初期化が必要なら呼ばれること

### 完了条件

- `TypeInference` がまだ空でも `ASTInserter` 単体で動作確認できる
- 参照シフトのルールがテストで固定されている
- 「壊れた原因が挿入か推論か」を切り分けられる

## フェーズ 3: TypeInference の最小内部型表現を作る

### 目標

AST 表現とは別に、推論しやすい内部型表現を用意します。
最初から高機能にせず、必要最低限から始めるのが大事です。

### 最初に持つとよい型

- `Unknown`
- `Named`
- `Ref`
- `MutRef`
- `Func`
- `Self`

後から追加してよいもの:

- `Generic`
- `Abs`
- 型変数
- 制約情報

### 方針

AST では `type_bw_&mut` のように分かれていても、内部では `Ref(isMut, inner)` のように寄せたほうが扱いやすいです。
内部表現は parser の都合ではなく、推論器の扱いやすさで決めてください。

### 推奨する補助関数

- AST 型ノードから内部型へ変換する関数
- 内部型を文字列表示する関数
- 2 つの型をマージする関数
- 「不明なら Unknown」を返す安全な解決関数

### まだやらないこと

- AST への書き戻し
- dedup
- 高度な総称型解決
- 最適化用キャッシュの作り込み

## フェーズ 4: 読み取り専用の宣言収集パスを作る

### 目標

この段階では AST を一切変更せず、推論に必要な宣言情報だけを集めます。
「何がどこに定義されているか」を知るのが目的です。

### 収集対象

- frame 名
- field 名と型注釈
- function 名
- impl 内 function の所属先
- parameter 名と型注釈
- 明示的 return 型

### 用意すると便利なテーブル

- `nodeRef -> 宣言情報`
- `function name -> 関数情報`
- `qualified name -> 関数情報`
- `frame name -> frame 情報`

### ここで大事なこと

このフェーズではまだ推論しなくてよいです。
return 型が未指定なら未指定のまま持てば十分です。
まずは情報を正しく引けることを優先してください。

### 完了条件

- 関数名から引数型と明示 return 型を引ける
- impl 内メソッドから self 相当の所属型を逆引きできる
- field 参照に必要な frame 情報を取得できる

## フェーズ 5: 式単位の型推論を小さく追加する

### 基本方針

いきなり全ノード対応を目指さず、簡単で壊れにくいものから順に増やしてください。

### 推奨する実装順

1. リテラル
2. 型注釈付き `let`
3. `ident`
4. 単純な `call_func`
5. ブロック末尾式
6. `if`
7. 参照と可変参照
8. フィールドアクセス
9. メソッド呼び出し

### 先に対応しやすいケース

- 数値リテラル
- 真偽値リテラル
- 文字列リテラル
- 型注釈付き `let`
- 明示 return 型付き関数

これらは情報源が明確なので、推論器の骨組みを作るのに向いています。

### Unknown の扱い

未対応ケースに遭遇したら、無理に推測せず `Unknown` を返してください。
初期段階では「間違った具体型」より「安全な不明」のほうがずっとましです。

### merge 規則

複数経路から型が集まる場合は、マージ規則を 1 箇所にまとめてください。

例:

- `Unknown` と具体型なら具体型を優先する
- 同一型同士ならその型を返す
- 数値同士の昇格を許すか早めに決める
- 矛盾時に `Unknown` に落とすか左優先にするかを統一する

### スコープ管理

最低限、次の情報は扱える必要があります。

- 関数引数
- `let` 束縛
- ブロック内シャドーイング
- `self` 相当の暗黙引数

可能なら `SimpleScope.fs` と責務が衝突しないように、推論器内の環境表現は薄く保つのがよいです。

## フェーズ 6: AST への型書き戻しを実装する

### 目標

推論した型を `FlatAST` に反映し、後続処理から参照できるようにします。

### 大原則

「何型かを決める処理」と「どこへどう挿入するか」は分離してください。
推論ロジックと AST 書き換えロジックを同じ関数に入れると保守が急激に難しくなります。

### 推奨する流れ

1. 書き戻し対象ノードを列挙する
2. 各ノードについて必要な型を計算する
3. 内部型を型ノード列へ変換する
4. `ASTInserter` で挿入する
5. 元ノードの `Data` を書き換える
6. 最後に `commit` する

### 書き戻し対象の候補

- 型注釈のない `let`
- return 型未指定の `func`
- return 型未指定の `func_impl`
- 必要なら `operand_func`

### 実装上の注意

- 挿入位置の決定規則を固定する
- 同じ型を複数回生成しても、最初は気にしすぎない
- 挿入でインデックスがずれるため、基本は後ろから処理する
- 挿入後のインデックス補正式を明文化する

### 共有と重複排除

初期段階では型ノードの共有や dedup を無理に入れなくて構いません。
まずは正しい AST が生成されることを優先してください。

## フェーズ 7: 最適化と整理

ここは最後にやるフェーズです。
先に入れると不具合の原因が見えにくくなります。

候補:

- 同一型ノードの重複排除
- 推論キャッシュ
- 関数 return 型のメモ化
- frame 継承やメンバー探索の高速化

最適化は、正しさが固まったあとで必要になってから入れるほうが安全です。

## 推奨するファイル内構成

### ASTInserter.fs

次の順番だと読みやすいです。

1. private helper
2. 参照更新関数
3. 挿入処理
4. 置換処理
5. commit
6. getter

### TypeInference.fs

次の順番が保守しやすいです。

1. 内部型定義
2. 小さな変換 helper
3. AST 参照 helper
4. 宣言収集
5. 式推論
6. 型ノード生成
7. AST 書き戻し
8. public member

## 各段階の確認ポイント

### ASTInserter 実装後

- 参照が壊れていないか
- 先頭・中間・末尾で一貫した挙動か
- 空挿入が安全か

### 宣言収集実装後

- 関数一覧が正しく集まるか
- frame と field の対応が正しいか
- impl 内関数が所属型を取得できるか

### 基本推論実装後

- リテラルの型が安定しているか
- `let` 束縛型を再利用できるか
- 単純関数の return 型が求まるか

### 書き戻し実装後

- 推論前後で AST 構造が大きく壊れていないか
- 型ノード参照が正しい位置を向いているか
- 複数連続挿入でも整合性が保たれるか

## 失敗しやすいポイント

### 1. 読み取りと書き込みを同じ関数でやる

一番壊れやすいです。
推論中に AST を変更すると、後続の解析が変更後状態を見てしまい、バグの再現性が落ちます。

### 2. Unknown の扱いが曖昧

未対応ケースを雑に具体型へ寄せると、誤った型情報が確定してしまいます。
Unknown は弱いですが、安全側です。

### 3. 参照シフトを複数箇所でやる

`ASTInserter` 以外でも参照番号を触り始めると整合性が崩れやすくなります。
参照更新は 1 箇所に閉じ込めるべきです。

### 4. 早い段階で dedup まで入れる

重複排除は便利ですが、書き戻しロジックが安定する前に入れると不具合の温床になります。
最初は重複を許して構いません。

## 最小マイルストーン

### マイルストーン 1

`ASTInserter` が完成し、挿入と置換がテストで保証される

### マイルストーン 2

`TypeInference` が frame / func / let の基本情報を読める

### マイルストーン 3

リテラル、ident、単純 call の推論ができる

### マイルストーン 4

単純関数の return 型を推論できる

### マイルストーン 5

型注釈なし `let` に型を書き戻せる

### マイルストーン 6

関数 return 型を AST に反映できる

### マイルストーン 7

`Self`、メソッド、フィールドアクセス、参照系を扱える

## public surface の案

再実装中も、外から見える API は小さく保つのがよいです。

```fsharp
type ASTInserter(fast: FlatAST) =
    member insert: int -> ASTNode -> unit
    member insertMany: int -> ASTNode list -> unit
    member replace: int -> ASTNode -> unit
    member commit: unit -> unit
    member getAST: unit -> ASTNode array

type TypeInference(fast: FlatAST) =
    member init: unit -> unit
    member materializeTypes: unit -> unit
    member getFlatAST: unit -> FlatAST
```

ただし今は `TypeInference` を空型にしているため、再導入時は `Program.fs` など呼び出し側も一緒に戻す必要があります。

## 実装時のルール

- 大きい関数を 1 本書かない
- フェーズごとに確認できる粒度で進める
- parser サンプルを固定して回帰確認する
- 未対応ケースは `Unknown` に倒す
- 最適化より先に正しさを固める
- AST 書き換えは後ろから処理することを基本にする

## 最後に

この再実装でいちばん大切なのは、機能数よりも「壊れ方が分かる構造」にすることです。
そのために、

- 読む処理
- 推論する処理
- 書き戻す処理

の 3 つは必ず分離してください。

この分離を守れば、途中で仕様変更が入っても修正箇所を限定できます。
逆にここが混ざると、型推論の機能追加がそのまま AST 破壊につながります。
