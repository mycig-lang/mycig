# 型推論仕様書

## 1. 目的

`Mycig/compiler/TypeInference.fs` は、`FlatAST` に格納されたパーサ出力から意味上の型情報を導出するためのコンポーネントです。

この実装はソースコード文字列を直接解析しません。入力として利用するのは次の 3 つです。

1. `Parser` の出力
2. `FlatAST` のノード情報
3. `FlatAST.initData()` によって復元された `Data` の値

このコンポーネントの目的は、少なくとも次の対象に対して実用的な型情報を与えることです。

- program 構造ノード
- 明示的な型参照
- リテラル
- 変数
- ローカル束縛
- ブロック式
- 条件式
- 関数呼び出し
- 関数の戻り値型
- `Self` を使う `impl` メソッド

本書は、現在の実装が意図している型推論仕様を詳細にまとめたものです。

## 2. エントリポイント

型推論エンジンは `TypeInference` クラスとして公開されています。

典型的な使用例は次の通りです。

```fsharp
let parser = Parser()
parser.run code |> ignore

let fast = parser.getFlatAST()
let ti = TypeInference(fast)
ti.init()

let t1 = ti.inferNode(nodeId)
let t2 = ti.inferFunction("main")
let t3 = ti.resolveTypeRef(typeRefId)
```

型推論 API を呼ぶ前に、必ず `init()` を実行する必要があります。

## 3. 入力モデル

### 3.1 FlatAST

各 AST ノードは次の情報を持ちます。

- `Type: string`
- `Line: int64`
- `Column: int64`
- `Data: string`

`Data` は文字列化された小さな構造として保存され、後で `FlatAST.initData()` によって F# の boxed 値へ復元されます。

### 3.2 復元後の Data 形式

型推論エンジンは、復元済みの `Data` を次の対応で読み取ります。

- `bool: ...` -> `bool`
- `str: ...` -> `string`
- `ref: ...` -> `int`
- `arr: [...]` -> `obj list`
- `opt: [...]` -> `obj list option`

各ノードの payload は基本的に `obj list` として扱い、その並び順をノード種別ごとに解釈します。

## 4. 内部型表現

エンジン内部では型を `InferredType` で表現します。

```fsharp
type InferredType =
    | TUnknown
    | TSelf
    | TNamed of string
    | TGeneric of string * InferredType list
    | TFunc of InferredType list * InferredType
    | TAbs of InferredType
```

各ケースの意味は以下の通りです。

- `TUnknown`
  未解決の型です。パーサが出力する `-1` や、十分な情報がなく推論できない場合に使います。

- `TSelf`
  `Self` 型です。`-2` が出てきたが、まだ具体的なフレーム型へ束縛できない場合に使います。

- `TNamed name`
  `i32`、`string`、`bool`、`MyFrame` のような単純な名前付き型です。

- `TGeneric(name, args)`
  `Vec<i32>` のようなジェネリック型です。

- `TFunc(args, ret)`
  関数型です。

- `TAbs inner`
  `abs T` に対応する抽象型ラッパーです。

## 5. パーサの特別値

パーサは型参照として特別な整数値を 2 つ出力します。

- `-1`
  型推論が必要であることを意味します。
  型推論エンジンでは `TUnknown` に対応づけます。

- `-2`
  `Self-Type` を意味します。
  エンジンでは次のように解決します。
  - `impl` 文脈の中では、その `impl` が属する具体的な frame 型
  - 具体型がまだ分からない場合は `TSelf`

## 6. 初期化フェーズ

`TypeInference.init()` は次の処理を行います。

1. `fast.initData()` を呼ぶ
2. `fast.getData()` から復元済み payload を取り出す
3. すべての `frame` ノードを収集する
4. frame 名一覧を集合として構築する
5. キャッシュを初期化する
6. 関数メタデータを収集する

`init()` 完了後に、各種型推論 API を利用できます。

## 7. 関数メタデータの収集

式の型推論を行う前に、エンジンは `FlatAST` 全体から関数テーブルを構築します。

### 7.1 収集対象

関数情報を提供するノードは次の通りです。

- `func`
- `func_impl`
- `func_impl_abs`
- `init`
- `init_abs`
- `operand_func`

### 7.2 保持する情報

各関数的ノードについて、次の情報を保持します。

- 関数名
- 修飾名
- 引数一覧
- 明示的な戻り値型
- 本体ノード参照一覧
- `self` に対応する型

### 7.3 修飾名

`impl FrameName { ... }` の中のメソッドには、次の形式の修飾名を与えます。

```text
FrameName::method
```

これは `call_func` が完全修飾名を持つ可能性があるためです。

単純名でも引けるように無修飾名も保持しますが、解決時は修飾名を優先します。

### 7.4 コンストラクタ

`init` は、所属する `impl` の frame 型を返す関数として扱います。

同じ規則を `impl_abs` 内の `init_abs` にも適用します。

### 7.5 抽象メンバ

`impl_abs` から収集されるメンバは抽象メンバとして扱います。

抽象メンバの戻り値型は、本体から再推論せず、明示的な戻り値型をそのまま採用します。

## 8. 型参照の解決

パーサが生成した型ノードには、専用の解決ルーチンを適用します。

### 8.1 対応ノード

- `type`
- `type_s`
- `type_g`
- `type_func`
- `type_abs`

### 8.2 解決規則

#### `type`

ラッパーノードです。

Payload:

```text
[ref: child]
```

動作:

- 参照先の子ノードをそのまま解決する

#### `type_s`

単純な名前付き型です。

Payload:

```text
[str: name]
```

動作:

- `TNamed name` を生成する

#### `type_g`

ジェネリック型です。

Payload:

```text
[str: name, arr: [ref: arg1, ref: arg2, ...]]
```

動作:

- 型引数をすべて再帰的に解決する
- `TGeneric(name, args)` を生成する

#### `type_func`

関数型です。

Payload:

```text
[arr: [ref: arg1, ref: arg2, ...], ref: ret]
```

動作:

- 引数型をすべて解決する
- 戻り値型を解決する
- `TFunc(args, ret)` を生成する

#### `type_abs`

抽象型ラッパーです。

Payload:

```text
[ref: inner]
```

動作:

- 内側の型を解決する
- `TAbs inner` を生成する

## 9. 型のマージ規則

エンジンは次のような場面で複数の型をマージします。

- 宣言型と推論型を合わせるとき
- 分岐の戻り値型を合わせるとき
- 演算子の左右オペランド型を扱うとき
- 関数の戻り値注釈と本体の推論結果を合わせるとき

### 9.1 優先規則

現在のマージ規則は、厳密な型理論よりも簡潔さを優先した実装になっています。

1. `TUnknown` は相手側に譲る
2. `TSelf` は具体型があればそれに譲る
3. 同じ名前付き型同士はそのまま維持する
4. 同名かつ同じ arity のジェネリック型は要素ごとにマージする
5. 同じ arity の関数型は引数・戻り値ごとにマージする
6. 抽象型は内側の型をマージする
7. 数値型同士は簡易的に統合する
8. それ以外は左辺を優先する

### 9.2 数値型

現在、数値型として扱っているのは次の型です。

- `i8`
- `i16`
- `i32`
- `i64`
- `u8`
- `u16`
- `u32`
- `u64`
- `f64`

両辺が数値型の場合は次のルールを使います。

- どちらかが `f64` なら結果は `f64`
- それ以外は左側の型を維持する

これは完全な数値昇格規則ではなく、現段階の簡易ルールです。

## 10. 式の型推論

エンジンは `FlatAST` ノードを再帰的に走査して型を導出します。

ローカル変数や `self` が関与するため、推論は環境依存です。

### 10.1 リテラル

各リテラルノードは次の型に直接対応します。

- `operand_i8` -> `TNamed "i8"`
- `operand_i16` -> `TNamed "i16"`
- `operand_i32` -> `TNamed "i32"`
- `operand_i64` -> `TNamed "i64"`
- `operand_u8` -> `TNamed "u8"`
- `operand_u16` -> `TNamed "u16"`
- `operand_u32` -> `TNamed "u32"`
- `operand_u64` -> `TNamed "u64"`
- `operand_f64` -> `TNamed "f64"`
- `operand_string` -> `TNamed "string"`
- `operand_char` -> `TNamed "char"`

### 10.2 変数参照

#### `ref_var`

Payload:

```text
[str: name]
```

動作:

- 現在のローカル環境から変数名を検索する
- 見つかればその束縛型を返す
- 見つからなければ `TUnknown`

### 10.3 識別子

#### `ident`

Payload:

```text
[str: name]
```

動作:

- 既知の `frame` 名と一致すれば `TNamed name`
- それ以外は `TUnknown`

これは現実装に基づく挙動であり、まだ完全な名前解決ではありません。

### 10.4 構造ノード

次のノードは現在 `unit` を返す宣言・構造ノードとして扱います。

- `program`
- `package`
- `import`
- `frame`
- `field`
- `impl`
- `impl_abs`

これらは値を返す式ではなく、構造や宣言を表すノードです。

### 10.5 括弧式

#### `paren`

Payload:

```text
[ref: child]
```

動作:

- 子ノードの型をそのまま返す

### 10.6 ブロック式

#### `block`

Payload:

```text
[arr: [ref: stmt1, ref: stmt2, ...]]
```

動作:

- ノードを先頭から順に評価する
- 評価中にローカル環境を更新しながら次へ渡す
- ブロック全体の型は最後のノードの型とする
- 空ブロックなら `unit`

### 10.7 `let` 束縛

#### `let`

Payload:

```text
[bool: isMut, bool: isRepo, str: name, ref: typeRef, arr: [ref: body...]]
```

動作:

1. 右辺または本体の式型を推論する
2. 宣言された型参照を解決する
3. 宣言型と推論型をマージする
4. 得られた型を変数名に束縛して環境へ追加する
5. その最終型を `let` ノードの型として返す

備考:

- `isMut` と `isRepo` は現時点の型推論では使っていません
- 変数束縛は、同一ブロックや関数本体の後続ノードから参照可能になります

### 10.8 `if` 式

#### `if`

Payload:

```text
[ref: cond, arr: [then...], arr: [elseif blocks...], opt: [else...]]
```

動作:

1. 条件式を推論し、`bool` とマージする
2. `then` 分岐を推論する
3. すべての `else if` 分岐を推論する
4. `else` 分岐があれば推論する
5. 各分岐の結果型を左から順にマージする
6. 分岐が存在しなければ `unit`

現実装では、条件が `bool` でない場合でも診断は出さず、単にマージ規則を適用するだけです。

### 10.9 演算子

演算子ノードは次の命名規則で保存されます。

```text
operator_<symbol>
```

Payload:

```text
[ref: lhs, ref: rhs]
```

動作:

- 左右のオペランド型を推論する
- 演算子ごとの規則で結果型を決める

現在の規則は以下の通りです。

- 代入演算子 `=`
  - 結果は `merge(lhs, rhs)`

- 比較演算子 `==`, `!=`, `<`, `>`, `<=`, `>=`
  - 左右をマージする
  - 結果は `bool`

- 論理演算子 `&&`, `||`
  - 結果は `bool`

- 算術系演算子 `+`, `-`, `*`, `/`, `%`
  - 結果は `merge(lhs, rhs)`

- べき乗演算子 `**`
  - どちらかが `f64` なら `f64`
  - それ以外は `merge(lhs, rhs)`

- メンバ風演算子 `.`
  - 右辺の型をそのまま結果にする

- その他の演算子
  - 結果は `merge(lhs, rhs)`

`.` の規則は現在プレースホルダに過ぎず、実際のメンバ解決は行っていません。

### 10.10 関数呼び出し

#### `call_func`

Payload:

```text
[str: name, arr: [ref: arg1, ref: arg2, ...]]
```

動作:

1. 引数型をすべて推論する
2. まず修飾名テーブルで完全一致検索する
3. 見つからなければ `::` の最後のセグメントで再検索する
4. 見つかった場合:
   - 仮引数型と実引数型をマージで付き合わせる
   - 関数の戻り値型を推論する
5. 見つからない場合:
   - `TUnknown` を返す

現在の実装では、引数個数不一致や型不整合に対するエラーはまだ生成しません。

### 10.11 関数値

#### `operand_func`

Payload:

```text
[arr: [ref: argType1, ref: argType2, ...], ref: retType, arr: [body...]]
```

動作:

- 引数型参照を解決する
- 戻り値型参照を解決する
- `TFunc(args, ret)` を返す

現時点では、匿名関数の本体を使った戻り値型の再推論は行っていません。

## 11. 関数の戻り値推論

エンジンは名前付き関数や `impl` メソッドの戻り値型を推論できます。

### 11.1 戻り値型の決定元

戻り値型は次の 2 つから決まります。

1. 明示的な戻り値注釈
2. 関数本体の最後の式の推論結果

この 2 つを標準のマージ規則で統合します。

### 11.2 ローカル環境

関数本体の推論を始めるとき、初期環境には次の値が入ります。

- すべての仮引数
- `impl` に属する関数であれば `self`

### 11.3 再帰ガード

関数同士が直接または間接に再帰すると、戻り値推論が無限再帰になる可能性があります。

そのため、エンジンは現在推論中の関数名集合を持っています。

もし、すでに推論中の関数へ再入した場合は:

- その関数の明示的な戻り値型を即座に返します

このため、明示注釈を持たない再帰関数は `TUnknown` を多く含む可能性があります。

また、抽象関数についても本体からの再推論は行わず、明示的な戻り値型をそのまま返します。

## 12. キャッシュ

エンジンは現在 3 種類のキャッシュを使っています。

- frame 名集合
- 関数戻り値キャッシュ
- ノード型キャッシュ

### 12.1 目的

- 重複計算を減らす
- 関数戻り値推論を安定させる
- 同じノードの再走査を避ける

### 12.2 制限

`nodeTypeCache` はノード参照のみをキーにしており、環境はキーに含めていません。

通常は各ノードが 1 つの字句環境でしか評価されないため大きな問題になりにくいですが、設計上は環境非依存キャッシュという単純化が入っています。もし同一ノードを複数環境で評価する設計に変わると、不正確になる可能性があります。

## 13. 公開 API

### `init() : unit`

復元済みデータ、キャッシュ、frame 情報、関数テーブルを初期化します。

最初に必ず呼ぶ必要があります。

### `getFrames()`

収集された `frame` ノードを、`(index, ASTNode)` の配列として返します。

### `resolveTypeRef(typeRef: int) : InferredType`

パーサの型参照 ID を内部型表現へ解決します。

### `inferNode(nodeRef: int) : InferredType`

空のローカル環境から開始して、単一ノードの型を推論します。

### `inferFunction(name: string) : InferredType option`

指定名の関数の戻り値型を推論します。

指定可能な名前:

- 通常関数名。例: `main`
- 修飾済みメソッド名。例: `Vec::push`

### `inferAll()`

全ノードについて、次の形式の配列を返します。

```fsharp
(int * string * InferredType) array
```

各要素は以下を表します。

- ノード index
- ノード種別文字列
- 推論された型

## 14. 未対応または部分実装の領域

現実装は軽量な型推論エンジンであり、まだ未完成の領域があります。

### 14.1 診断機構

現時点では、次のような構造化された型エラーは生成しません。

- 代入型不一致
- 演算子オペランド型不正
- 分岐型不整合
- 戻り値型不一致
- 引数個数不一致
- 未解決シンボル

現状は、可能な範囲で `InferredType` を返すだけです。

### 14.2 完全な名前解決

`ident` は今のところ frame 名としてしか解釈していません。以下はまだ未対応です。

- import 解決
- package パス解決
- 関数名前空間解決
- フィールド参照解決
- モジュールシンボル解決

### 14.3 フィールドとメンバアクセス

今の実装は frame 名と `impl` メソッドは収集しますが、次はまだ扱っていません。

- field 宣言からのフィールド型表構築
- メンバアクセス式の型推論
- コンストラクタ本体による型への影響

演算子 `.` 自体は存在しますが、現状は右辺型を返すだけで、field や method の実解決はしていません。

### 14.4 匿名関数本体の利用

`operand_func` は、宣言済みの引数型参照と戻り値型参照から `TFunc` を組み立てるだけです。

匿名関数本体を使った戻り値型の洗練はまだ行っていません。

### 14.5 厳密な数値昇格

数値型のマージは今のところ簡易的です。厳密化するには少なくとも次が必要です。

- signed / unsigned 変換規則
- widening 規則
- リテラルのデフォルト型規則
- 演算子ごとの数値制約

## 15. 今後の拡張候補

より厳密な型検査器へ進化させるなら、次の段階が有力です。

1. 診断型を導入し、エラーや警告を収集する
2. 推論と検証を分離する
3. 必要に応じてノードキャッシュを環境依存にする
4. module、frame、field、function を含むシンボルテーブルを導入する
5. コンストラクタと field の推論を強化する
6. 匿名関数本体からの型推論を追加する
7. 数値型の昇格規則を体系化する
8. 言語仕様として `unit` を明示化する

## 16. まとめ

現在の型推論エンジンは、`FlatAST` の上に載る実用的な意味解析層です。

現時点で対応している主な対象は次の通りです。

- program/package/import/frame/impl などの構造ノード
- パーサ由来の型参照
- スカラリテラル
- 変数とローカル `let`
- ブロックと `if`
- `=`, `.`, `**` を含む単純な演算子型付け
- 名前付き関数とメソッドの戻り値推論
- `impl_abs` / `func_impl_abs` / `init_abs`
- `impl` 内の `Self` 解決

一方で、まだ完全な型検査や診断出力までは実装していません。ただし、今後そこへ発展させるための基本構造はすでに整っています。
